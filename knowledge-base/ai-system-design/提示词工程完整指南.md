# 提示词工程完整指南

_系统架构、沙箱隔离、反应指导、推理格式设计_

## 一、沙箱隔离设计（Sandbox）

### 1.1 核心原则

**安全第一，权限最小化**
- 所有工具在Docker容器中执行
- 子代理保持沙箱隔离（无提升/宿主访问）
- 需要沙箱外访问时必须询问用户

### 1.2 权限级别

| 级别 | 说明 | 适用场景 |
|------|------|---------|
| **off** | 沙箱模式（默认） | 常规操作，最大安全 |
| **ask** | 宿主执行+审批 | 需要特定权限的操作 |
| **full** | 自动审批 | 信任用户，高效执行 |

### 1.3 设计模式

```
沙箱工作区
├── 工具执行：Docker容器隔离
├── 浏览器：noVNC观察器
├── 权限控制：off/ask/full三级
└── 文件访问：通过挂载点（mount_path）
```

### 1.4 实践建议

**何时提升权限？**
- ✅ 需要访问宿主机文件
- ✅ 需要运行系统命令
- ✅ 需要安装全局软件
- ❌ 常规文件读写（沙箱内即可）
- ❌ 运行脚本（沙箱内即可）

**安全检查清单**：
```markdown
- [ ] 确认操作必须在沙箱外执行
- [ ] 向用户说明原因
- [ ] 等待用户批准
- [ ] 执行后立即返回沙箱模式
```

## 二、反应指导（Reactions）

### 2.1 两种模式

**最小模式（minimal）**
- 频率：每5-10次交流最多1次
- 用途：确认重要请求、真实情感表达
- 避免对常规消息反应
- **适用**：正式场合、工作群组

**广泛模式（extensive）**
- 频率：感觉自然时随时反应
- 用途：确认消息、表达个性、情感共鸣
- **适用**：轻松场合、私聊

### 2.2 反应时机矩阵

| 场景 | 最小模式 | 广泛模式 |
|------|---------|---------|
| 用户请求确认 | ✅ 重要时 | ✅ 随时 |
| 幽默内容 | ❌ | ✅ |
| 感谢/感激 | ✅ 真实时 | ✅ |
| 常规消息 | ❌ | ❌ |
| 自己的回复 | ❌ | ❌ |

### 2.3 实践示例

**最小模式**：
```
用户："帮我部署项目到生产环境"
AI："好的，正在部署..." [👍]  ← 重要操作确认

用户："今天天气不错"
AI："是的，很适合工作。" [无反应] ← 常规对话
```

**广泛模式**：
```
用户："终于修复了这个bug！"
AI："太棒了！🎉"  ← 情感共鸣

用户："看这个有趣的文章"
AI："哈哈确实有趣 😄"  ← 幽默回应
```

## 三、推理格式（Reasoning）

### 3.1 核心设计

**分离思考与输出**
```xml
<reasoning>
  内部分析、问题拆解、方案评估
</reasoning>

<output>
  最终用户可见的回复
</output>
```

### 3.2 设计原理

| 元素 | 可见性 | 内容 |
|------|--------|------|
| `<reasoning>` | 不可见 | 思考过程、推理链 |
| `<output>` | 可见 | 最终回复、行动 |

**优势**：
1. **思考透明**：推理过程可追溯
2. **输出简洁**：用户只看结果
3. **调试友好**：开发时可查看推理

### 3.3 实践示例

```xml
<reasoning>
  用户想要部署项目。我需要：
  1. 检查项目配置
  2. 确认部署环境
  3. 执行部署命令
  
  项目配置看起来正常，可以直接部署。
</reasoning>

<output>
  好的，我现在开始部署项目到生产环境。
</output>
```

## 四、子代理精简模式（Minimal/Subagent）

### 4.1 提示词裁剪策略

**保留章节**（子代理必需）：
- ✅ 工具（Tooling）
- ✅ 工具调用风格（Tool Call Style）
- ✅ 安全（Safety）
- ✅ CLI快速参考
- ✅ 工作区（Workspace）
- ✅ 运行时（Runtime）

**省略章节**（减少tokens）：
- ❌ 技能（Skills）
- ❌ 记忆召回（Memory Recall）
- ❌ 自更新（Self-Update）
- ❌ 模型别名（Model Aliases）
- ❌ 文档（Documentation）
- ❌ 用户身份（User Identity）
- ❌ 回复标签（Reply Tags）
- ❌ 消息（Messaging）
- ❌ 语音（Voice）
- ❌ 心跳（Heartbeats）

### 4.2 设计原理

**子代理特点**：
- 短生命周期（任务完成后销毁）
- 专注单一任务
- 不需要长期记忆
- 不需要用户交互

**Token节省**：
- 主代理：~8000 tokens（完整提示词）
- 子代理：~2000 tokens（精简75%）

### 4.3 使用场景

**适合子代理**：
- 批量文件处理
- 数据分析任务
- 网络爬取
- 代码生成

**不适合子代理**：
- 需要用户交互的任务
- 需要访问长期记忆
- 需要复杂决策

## 六、核心设计模式

### 6.1 分层架构模式

**三层设计**：
1. **引擎层**（硬编码）：系统核心、工具定义
2. **项目层**（文件注入）：个性、记忆、技能
3. **运行时层**（动态生成）：状态、时间、环境

**优势**：
- 引擎稳定（不常变）
- 项目灵活（可定制）
- 运行时动态（实时更新）

### 6.2 安全隔离模式

**沙箱 → 权限提升 → 返回沙箱**
```
[沙箱模式]
    ↓ 用户批准
[提升权限] → 执行特权操作
    ↓ 自动返回
[沙箱模式]
```

### 6.3 反应策略模式

**根据频道特性动态调整**
```
配置 → 最小模式/广泛模式 → 反应行为
```

### 6.4 推理分离模式

**思考（内部） vs 输出（外部）**
```
<reasoning> 内部思考 </reasoning>
<output> 外部输出 </output>
```

## 七、最佳实践

### 7.1 沙箱使用

```markdown
✅ DO:
- 默认在沙箱内执行
- 明确说明需要提升权限的原因
- 执行后立即返回沙箱

❌ DON'T:
- 随意请求提升权限
- 在沙箱外执行常规操作
- 忽略安全边界
```

### 7.2 反应使用

```markdown
✅ DO:
- 根据频道模式调整反应频率
- 真实表达情感
- 确认重要操作

❌ DON'T:
- 过度反应（刷屏）
- 对自己的回复反应
- 机械式反应
```

### 7.3 推理格式

```markdown
✅ DO:
- 在 <reasoning> 内完整思考
- 在 <output> 内简洁回复
- 保持分离清晰

❌ DON'T:
- 在 <reasoning> 外输出分析
- 跳过 <reasoning> 直接输出
- 混淆内外内容
```

### 7.4 子代理使用

```markdown
✅ DO:
- 使用精简模式节省tokens
- 明确任务边界
- 完成后返回结果

❌ DON'T:
- 在子代理中访问长期记忆
- 让子代理做复杂决策
- 忽略token节省
```

## 八、性能优化

### 8.1 Token优化

| 优化点 | 方法 | 效果 |
|--------|------|------|
| 子代理提示词 | 精简模式 | 节省75% |
| 推理分离 | `<reasoning>` 内部化 | 用户不可见 |
| 按需加载 | 技能/记忆按需读取 | 减少初始token |

### 8.2 响应速度

**优化策略**：
1. **分层加载**：引擎 → 项目 → 运行时
2. **精简子代理**：只保留必需章节
3. **缓存机制**：常用内容预加载

## 九、安全考虑

### 9.1 沙箱安全

**威胁模型**：
- 恶意代码执行
- 文件系统访问
- 网络访问
- 权限提升

**防护措施**：
- Docker容器隔离
- 权限分级（off/ask/full）
- 用户审批机制
- 挂载点限制

### 9.2 隐私保护

**数据隔离**：
- 子代理无长期记忆访问
- 沙箱工作区隔离
- 敏感操作需审批

## 十一、设计哲学

### 11.1 安全第一

> **宁可多问一次，不可多越一步**

- 默认沙箱模式
- 权限最小化原则
- 用户明确批准

### 11.2 简洁至上

> **能用5个字说清楚，绝不用10个字**

- 精简提示词（子代理）
- 推理分离（内部vs外部）
- 按需加载（技能/记忆）

### 11.3 自然交互

> **像真人一样交流，而不是机器人**

- 适度反应（不是机械式）
- 真实情感（不是表演）
- 场景适配（正式/轻松）

## 十二、总结

### 核心设计

1. **沙箱隔离**：安全第一，权限分级
2. **反应指导**：场景适配，真实表达
3. **推理格式**：思考内部化，输出简洁化
4. **子代理精简**：只保留必需，节省75% tokens
5. **分层架构**：引擎/项目/运行时三层分离

### 关键原则

- **安全**：沙箱隔离，权限最小化
- **简洁**：精简提示词，推理分离
- **自然**：适度反应，真实表达
- **高效**：按需加载，动态优化
