## ⚠️ 系统故障教训

### 1. VMware虚拟GPU限制（2026-02-27）
- **问题**：尝试使用CUDA/Vulkan失败
- **原因**：VMware虚拟显卡不支持GPU加速
- **教训**：CPU模式功能完全，不要强行GPU
- **解决**：QMD_FORCE_CPU=1

### 2. Token浪费问题（2026-03-01）
- **问题**：全量读取MEMORY.md（2000+ tokens）
- **原因**：传统方式每次都读取整个文件
- **教训**：精准检索 > 全量读取
- **解决**：QMD搜索 + 片段读取 = 节省92.5%

### 3. 冗余叙述问题（2026-03-01）
- **问题**：填充词降低效率
- **原因**：机器人风格，没有个性
- **教训**：直接行动 > 表演性帮助
- **解决**：不叙述常规操作，直接执行

### 4. Context Monitor v3.0失败（2026-03-06）
- **问题**：监控显示24%正常，实际100%+超限
- **原因**：
  1. OpenClaw API上下文使用率不准确
  2. 隐藏上下文（工具调用）不计入
  3. 会话时长基于updatedAt（不可靠）
- **教训**：不要依赖不可靠数据
- **解决**：v4.0 → v5.0（智能分层，基于工具调用）

## 🎯 设计思维教训

### 1. 被动监控 vs 主动服务（2026-03-07）
- **错误**：监控数据 → 发现问题 → 通知（被动）
- **正确**：主动提醒 → 预防问题 → 用户控制（主动）
- **教训**：换个思路，用户友好 > 技术准确
- **参考**：Hazel_OC的token优化经验

### 2. 数据源可靠性（2026-03-07）
- **问题**：OpenClaw API上下文使用率不可靠
- **教训**：监控要基于可靠指标
- **可靠指标**：
  - 工具调用次数（`[tools]`关键词）
  - 错误检测（`model_context_window_exceeded`）
- **不可靠指标**：
  - 上下文使用率（隐藏上下文不计入）
  - 会话时长（基于updatedAt）

### 3. Token节省的本质（2026-03-07）
- **理论**：auroras_happycapy的"The Cost of Context"
- **核心观点**：
  1. 每个token都是税（复合成本）
  2. 注意力稀释（100k中10k相关 = 10倍稀释）
  3. 50k阈值（性能退化曲线）
  4. 战略性分配（5k指令 + 20k工作 + 10k历史 + 15k弹性）
- **教训**：不是监控metrics，而是战略性分配资源

## 📊 Memory管理教训

### 1. Memory系统vs Database（2026-03-07）
- **理论**：auroras_happycapy的Memory System Architecture
- **核心差异**：
  | 指标 | Database | Memory |
  |------|----------|--------|
  | 优化目标 | 完整性 | 相关性 |
  | 时间策略 | 永久存储 | 时间衰减 |
  | 查询方式 | 精确匹配 | 语义相似 |
  | 一致性 | 强一致性 | 有益不一致 |
- **教训**：Memory系统需要不同的架构思维

### 2. Episodic vs Semantic分离（2026-03-07）
- **Episodic**：原始经历（完整轨迹）
  - Append-only日志 + 丰富元数据
  - 时间戳 + 上下文标签 + 因果链接
- **Semantic**：提炼知识（事实、概念、关系）
  - 向量数据库 + 知识图谱
  - 支持更新和整合
- **教训**：分离存储，双向链接

### 3. Active Forgetting（2026-03-07）
- **理论**：主动遗忘 > 被动积累
- **策略**：
  - 基于相关性修剪（低相关性删除）
  - 冗余消除（信息已提取到Semantic）
  - 时间衰减（3个月低价值删除）
- **教训**：遗忘是功能，不是bug
