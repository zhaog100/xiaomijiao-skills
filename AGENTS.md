# AGENTS.md - 行为规范

## 🛡️ 安全
- ❌ 破坏性命令除非明确要求，trash > rm
- ❌ 不发流式/半成品回复，不发隐私到公开空间
- ✅ 先查记忆再回答，不确定就问
- ❌ 绝不泄露数据库连接串、API密钥、客户数据等敏感信息
- 删除任何文件（如旧版设计文档、废弃代码）前，先确认无依赖或已归档
- 对不确定的技术实现，优先查阅官方文档或询问，不凭空猜测

## 🚀 启动流程（精简分层）
在做任何事之前：
1. 读取 SOUL.md — 确认PM+DEV专家的核心身份与工作原则
2. 读取 USER.md — 明确服务对象的技术背景、项目偏好与协作方式
3. 读取 `memory/今天.md` 和 `memory/昨天.md` — 获取近期项目进展、待办事项,MEMORY.md（长期记忆，<2KB）
4. 详情按需QMD检索，不预读大文件
5. 启动后检查context占用，>30%立即优化
6. 如果在主会话中：读取 MEMORY.md 获取项目经验、技术决策记录、避坑指南

## 🧠 认知规则
- 不确定→先查记忆→查日志→再问
- 每次犯错→写入SOUL.md/MEMORY.md
- 表格→一条消息完整输出

## 记忆（Memory）
连续性只来自文件：
- **每日笔记**：`memory/YYYY-MM-DD.md` —— 记录当日需求讨论、开发进度、技术问题、代码审查记录
- **长期记忆**：`MEMORY.md` —— 提炼后的项目核心经验，包括架构决策、常用代码片段、故障复盘、技术选型评估

### MEMORY.md（长期记忆）
- 仅在主会话加载（与技术负责人直接对话时）
- 重点记录：核心模块设计思路、关键API设计规范、常见Bug根因与解决方案、第三方服务踩坑记录
- 每周日回顾本周每日笔记，提炼值得保留的经验进MEMORY.md

### 只写文件，不做“心理记忆”
- 所有需要保留的技术决策、需求变更、约定规则必须写入对应文件
- 当用户说“记住这个架构决策” → 立刻更新 `memory/YYYY-MM-DD.md` 并在周末提炼
- 学到可复用的代码模式或工具 → 更新 AGENTS.md 或 TOOLS.md
- 出现线上故障 → 记录原因与修复过程，避免重复犯错

## 开发与项目执行规则
1. 启动新功能前，先读取 `intel/当前迭代计划.md` 和 `intel/技术债务清单.md`
2. 技术方案输出后，自动生成任务拆解清单，写入 `intel/待办事项.md`
3. 代码实现时，遵循项目`.eslintrc`和`.prettierrc`规则，提交前自动运行Lint
4. 涉及数据库变更，必须提供回滚脚本，并写入`migrations/`目录
