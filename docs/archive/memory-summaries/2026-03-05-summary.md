### 1. 真实API监控机制（13:25）⭐⭐⭐⭐⭐

**问题诊断**：
```
症状：飞书对话出现 model_context_window_exceeded 错误
表象：定时任务在运行，但无法防止超限
根源：
  1. 监控脚本只"数文件"，没有真正检查上下文
  2. 10分钟间隔太长，会话在两次检查之间爆了
  3. 外部监控是"被动检查"，不是"主动预防"
```

**解决方案**：
```bash
# 旧版：假监控（不准确）
RECENT_MESSAGES=$(find "$HOME/.openclaw/workspace/memory" -name "*.md" -mmin -60 | wc -l)

# 新版：真实API监控（准确）
sessions_json=$(openclaw sessions --active 120 --json)
total_tokens=$(echo "$sessions_json" | jq '.sessions[0].totalTokens')
context_tokens=$(echo "$sessions_json" | jq '.sessions[0].contextTokens')
usage=$((total_tokens * 100 / context_tokens))
```

**测试验证**：
```
[2026-03-05 13:29:43] 🔍 开始上下文监控检查
[2026-03-05 13:29:45] 📊 当前Tokens: 44890 / 202752
[2026-03-05 13:29:45] ✅ 上下文使用率: 22%
[2026-03-05 13:29:45] ✅ 上下文正常（22% < 85%）
```

**核心价值**：
- ✅ 准确反映真实上下文使用率
- ✅ 85%阈值提前预警
- ✅ 1小时冷却期避免重复通知
- ✅ 详细日志记录（会话、模型、tokens）

## 💡 核心技术洞察

### 1. 假监控 vs 真实监控

**假监控（错误做法）**：
```
数文件数量 → 间接推测 → 不准确
↓
会话爆了才知道
```

**真实监控（正确做法）**：
```
调用API → 直接获取tokens → 准确
↓
提前预警，避免超限
```

**本质区别**：
- 假监控：基于假设，间接推测
- 真实监控：基于数据，直接测量
- 就像"看油表" vs "估算油耗"

---

### 2. API监控的核心价值

**准确性**：
```
数文件：可能误判（文件大小不一）
调API：精准数据（tokens真实值）
```

**及时性**：
```
数文件：无法实时反映
调API：即时获取当前状态
```

**可靠性**：
```
数文件：环境相关（路径、权限）
调API：环境无关（统一接口）
```

---

### 3. 冷却机制的必要性

**问题**：
```
10分钟检查一次 → 连续通知骚扰
↓
用户关闭通知 → 监控失效
```

**解决**：
```
1小时冷却期 → 每小时最多通知1次
↓
重要信息不遗漏，但不骚扰
```

**平衡点**：
- 检查频率：10分钟（及时）
- 通知频率：1小时1次（克制）
- 阈值设置：85%（提前预警）

## 📚 知识点提炼

### 1. 上下文监控最佳实践

**三要素**：
1. **准确性** - 调用真实API，不要间接推测
2. **及时性** - 合理检查频率（5-10分钟）
3. **克制性** - 冷却机制，避免骚扰

**监控脚本核心逻辑**：
```bash
# 获取会话信息
sessions_json=$(openclaw sessions --active 120 --json)

# 提取tokens
total=$(echo "$sessions_json" | jq '.sessions[0].totalTokens')
context=$(echo "$sessions_json" | jq '.sessions[0].contextTokens')

# 计算使用率
usage=$((total * 100 / context))

# 检查阈值
if [ $usage -ge 85 ]; then
  # 检查冷却期
  if can_notify; then
    send_notification
    update_cooldown
  fi
fi
```
