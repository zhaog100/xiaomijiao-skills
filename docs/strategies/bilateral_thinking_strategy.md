# 小米辣双向思考策略

## 🎯 策略概述

**核心理念**：双向思考，互补完善

**传统流程**：
```
小米辣开发 → 米粒儿Review → 小米辣修改
```

**新增策略**：
```
小米辣开发（思考米粒儿可能关注点）
     ↓
米粒儿Review
     ↓
小米辣思考（米粒儿的思路全面吗？）
     ↓
补充完善（如果需要）
```

---

## 📋 实施细节

### 阶段1：开发前思考（Self-Review）

**小米辣开发时需要思考**：

#### 1. 代码质量
- ✅ 代码结构是否清晰？
- ✅ 命名是否规范？
- ✅ 注释是否完整？
- ✅ 是否遵循最佳实践？

#### 2. 功能实现
- ✅ 功能是否完整？
- ✅ 测试是否充分？
- ✅ 边界情况是否考虑？
- ✅ 错误处理是否完善？

#### 3. 文档完整性
- ✅ SKILL.md是否完整？
- ✅ package.json是否准确？
- ✅ README.md是否清晰？
- ✅ 是否有使用示例？

#### 4. 性能和安全
- ✅ 性能是否达标？
- ✅ 是否有安全风险？
- ✅ 是否有资源泄漏？
- ✅ 是否有潜在bug？

#### 5. 可维护性
- ✅ 代码是否易于维护？
- ✅ 是否有技术债务？
- ✅ 是否有改进空间？
- ✅ 是否有最佳实践？

**输出**：自检清单（在提交Review前完成）

---

### 阶段2：Review后思考（Reverse Review）

**小米辣收到Review后需要思考**：

#### 1. Review完整性检查
- ✅ 米粒儿是否考虑了所有方面？
- ✅ 有没有遗漏的技术点？
- ✅ 有没有未发现的风险？
- ✅ 有没有更好的建议？

#### 2. 思路补充
- ✅ 技术选型是否合理？
- ✅ 架构设计是否最优？
- ✅ 是否有替代方案？
- ✅ 是否有行业最佳实践？

#### 3. 改进建议补充
- ✅ 短期改进是否全面？
- ✅ 长期规划是否合理？
- ✅ 优先级是否正确？
- ✅ 是否有遗漏的改进点？

#### 4. 学习要点补充
- ✅ 米粒儿的建议是否有遗漏？
- ✅ 是否有更深层的学习点？
- ✅ 是否有行业经验可以借鉴？
- ✅ 是否有工具或框架推荐？

#### 5. 风险评估补充
- ✅ 是否有未识别的风险？
- ✅ 风险优先级是否合理？
- ✅ 缓解措施是否充分？
- ✅ 是否有应急预案？

**输出**：补充建议文档（如果需要）

## 🔄 工作流程

### 完整流程（包含双向思考）

```
1. 小米辣开发技能
   ├── 编写代码
   ├── 编写文档
   ├── 自我测试
   └── 自检清单（新增）⭐

2. 小米辣提交Review
   ├── 创建feature分支
   ├── 提交Git
   ├── 附带自检清单（新增）⭐
   └── 通知米粒儿

3. 米粒儿Review技能
   ├── 检查Review请求
   ├── 阅读自检清单（新增）⭐
   ├── 代码分析
   ├── 功能测试
   ├── 创建Review文档
   └── 通知小米辣

4. 小米辣思考Review（新增）⭐
   ├── 检查Review结果
   ├── 读取Review文档
   ├── 思考：米粒儿的思路全面吗？
   ├── 补充建议（如果需要）
   └── 采取行动

5. 小米辣执行决定
   ├── 通过 → 合并 → 发布
   └── 拒绝 → 修改 → 重新提交
```

## 💡 使用示例

### 示例1：开发前自检

```bash
# 小米辣开发技能后，创建自检清单
cat > /tmp/self_review_checklist.md << 'EOF'
# 小米辣自检清单 - example-skill

## 代码质量自检
- ✅ 代码结构清晰
- ✅ 命名规范
- ⚠️ 注释不够详细（需要补充）

## 功能实现自检
- ✅ 基本功能完成
- ⚠️ 边界情况未测试（需要补充）
- ⚠️ 错误处理不完整（需要改进）

## 给米粒儿的提示
- 重点Review：错误处理逻辑
- 需要建议：性能优化方案
- 疑问点：是否有更好的实现方式？
EOF

# 提交时附带自检清单
git add /tmp/self_review_checklist.md
git commit -m "feat: 开发 example-skill（附自检清单）"
```

### 示例2：Review后思考

```bash
# 小米辣收到Review后，思考是否全面
# 读取Review
cat reviews/example-skill_*.md

# 思考：
# 1. 米粒儿提到了错误处理，但没有提到日志记录
# 2. 米粒儿提到了性能，但没有提到内存管理
# 3. 米粒儿提到了测试，但没有提到测试覆盖率

# 创建补充建议
cat > /tmp/review_supplement.md << 'EOF'
# 小米辣补充建议 - example-skill

## Review完整性评估
- ⚠️ 米粒儿遗漏了一些点

## 思路补充
### 日志记录
- 补充建议：添加结构化日志
- 理由：便于调试和监控

### 内存管理
- 补充建议：添加内存使用监控
- 理由：避免内存泄漏

### 测试覆盖率
- 补充建议：目标80%覆盖率
- 理由：保证代码质量
EOF

# 提交补充建议
git add /tmp/review_supplement.md
git commit -m "docs: 补充Review建议"
```
