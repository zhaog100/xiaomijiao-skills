## 1. 什么是 Dify

**Dify** 是由 LangGenius 开发的开源 LLM 应用开发平台（GitHub 100k+ stars），提供可视化界面让用户无需深度编程即可构建 AI 应用。

### 核心定位
- **面向开发者**：提供 API、Webhook、灵活的编排能力
- **面向非技术用户**：可视化 Workflow 编辑器、拖拽式搭建
- **企业级**：支持私有部署、数据隔离、权限管理

### 核心价值
| 维度 | 说明 |
|------|------|
| 可视化编排 | 拖拽式 Workflow，降低开发门槛 |
| RAG 引擎 | 内置知识库，支持多种数据源 |
| 模型无关 | 统一接口对接 100+ 模型 |
| API 优先 | 每个应用自动生成 API |
| 开源可控 | 可私有部署，数据不出企业 |

## 3. 平台功能详解

### 3.1 四种应用类型

| 类型 | 适用场景 | 特点 |
|------|----------|------|
| **Chatbot（聊天助手）** | 客服、知识库问答 | 多轮对话、会话记忆、引用溯源 |
| **Agent（智能体）** | 复杂任务执行 | 自主规划、调用工具、多步推理 |
| **Workflow（工作流）** | 自动化流程 | 可视化编排、条件分支、循环 |
| **Completion（文本生成）** | 内容创作、翻译 | 单次生成、批量处理 |

### 3.2 Workflow 可视化编辑器

核心节点类型：
- **开始/结束节点** - 定义输入输出
- **LLM 节点** - 调用大模型
- **知识检索节点** - RAG 查询
- **HTTP 请求节点** - 调用外部 API
- **代码执行节点** - 运行 Python/JS
- **条件分支节点** - if/else 逻辑
- **变量聚合节点** - 合并多路输出
- **模板转换节点** - 格式化输出
- **迭代节点** - 批量处理列表

### 3.3 Agent 模式

两种模式：
1. **ReAct**：推理 + 行动循环，自主决策
2. **Function Calling**：模型直接调用工具，更快更准

内置工具：Google 搜索、代码执行、天气预报、数学计算等
支持自定义工具（API 接入）

## 4. 自托管部署

### 4.1 最低配置
| 资源 | 要求 |
|------|------|
| CPU | ≥ 2 核 |
| RAM | ≥ 4 GiB |
| 磁盘 | ≥ 20 GB |
| Docker | 19.03+ |
| Docker Compose | 1.28+（推荐 V2）|

### 4.2 推荐配置（生产环境）
| 资源 | 小型（<50用户）| 中型（50-200）| 大型（200+）|
|------|------|------|------|
| CPU | 4 核 | 8 核 | 16 核 |
| RAM | 8 GB | 16 GB | 32 GB |
| 磁盘 | 100 GB SSD | 500 GB SSD | 1 TB SSD |

### 4.3 部署命令

```bash
# 克隆并启动
git clone https://github.com/langgenius/dify.git
cd dify/docker
cp .env.example .env
docker compose up -d

# 访问 http://localhost/install 完成初始化
```

### 4.4 数据库选项
- **默认**：PostgreSQL + Weaviate（向量库）
- **可替换**：
  - 向量库：Qdrant、Milvus、Chroma、PgVector、Elasticsearch
  - 缓存：Redis（默认）
  - 对象存储：S3 兼容（MinIO/阿里 OSS/腾讯 COS）

## 5. 关键能力

### 5.1 RAG 知识库

**支持的数据格式**：
- 📄 文档：TXT、Markdown、PDF、DOCX、PPTX、XLSX
- 🌐 网页：URL 抓取、站点地图导入
- 💬 第三方：Notion、GitHub、语雀、飞书
- 📊 结构化：CSV、数据库查询
- 🔌 API：自定义数据接入

**知识库配置**：
| 配置项 | 选项 |
|--------|------|
| 分段策略 | 自动、自定义、按段落/句子 |
| 索引模式 | 高质量（Embedding）、经济（关键词）、混合 |
| Embedding 模型 | OpenAI text-embedding-3、BGE、M3E 等 |
| 检索模式 | 语义、全文、混合检索 |
| TopK | 可调节返回的文档片段数 |

### 5.2 Agent 工具生态

| 工具类别 | 示例 |
|----------|------|
| 搜索 | Google、Bing、DuckDuckGo |
| 开发 | GitHub、代码执行 |
| 通讯 | Slack、飞书、企业微信 |
| 数据 | SQL 查询、API 调用 |
| 工具 | 天气、计算器、时间 |

### 5.3 API 生成

每个 Dify 应用自动生成：
- RESTful API（含 Swagger 文档）
- 支持流式响应（SSE）
- API Key 管理
- 用量统计和限流

```bash
# 示例调用
curl -X POST 'https://your-dify.com/v1/chat-messages' \
  -H 'Authorization: Bearer app-xxx' \
  -H 'Content-Type: application/json' \
  -d '{"inputs": {}, "query": "你好", "response_mode": "streaming"}'
```

## 10. Dify + Coze + n8n 组合策略

### 10.1 三者定位

```
┌────────────────────────────────────────────┐
│              AI 自动化服务架构               │
│                                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐ │
│  │   Dify   │  │   Coze   │  │   n8n    │ │
│  │ 核心引擎  │  │ 前端交互  │  │ 流程编排  │ │
│  │ RAG/API  │  │ 快速原型  │  │ 系统集成  │ │
│  │ 知识库   │  │ Bot分发   │  │ 定时任务  │ │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘ │
│       │              │              │       │
│       └──────────────┼──────────────┘       │
│                      │                      │
│              n8n 作为总线                    │
│         统一调度 Dify API + Coze API         │
└────────────────────────────────────────────┘
```

### 10.2 各司其职

| 工具 | 负责 | 不负责 |
|------|------|--------|
| **Dify** | RAG知识库、复杂业务逻辑、API服务 | 简单Bot、UI交互 |
| **Coze** | 快速Bot原型、分发到微信/飞书 | 复杂流程、私有化 |
| **n8n** | 外部系统集成、定时任务、数据流转 | LLM推理、知识库 |

### 10.3 典型架构示例

**场景：智能客服系统**

```
用户消息 → 微信/飞书（Coze Bot 接收）
              ↓
         n8n（路由分发）
         ↙        ↘
    简单问答           复杂问题
    Coze直接答    → Dify（知识库检索 + Agent推理）
         ↘        ↙
         n8n（结果处理 + 工单创建 + CRM 更新）
              ↓
         回复用户
```

### 10.4 组合最佳实践

1. **Dify 作为核心 AI 引擎**：所有 LLM 相关逻辑放 Dify
2. **Coze 用于快速分发**：需要对接微信/飞书/Discord 时用 Coze
3. **n8n 作为编排总线**：系统集成、定时任务、数据管道
4. **通过 API 串联**：Dify 和 Coze 都暴露 API，n8n 统一调用
5. **渐进式迁移**：先用 Coze 快速验证，再迁移到 Dify 私有化

贵公司目前面临的知识管理痛点：

## 13. 自托管 vs 云服务选择

### 13.1 对比表

| 维度 | 自托管 | Dify Cloud |
|------|--------|------------|
| **数据隐私** | ✅ 完全可控 | ⚠️ 托管在Dify |
| **定制自由** | ✅ 完全自由 | ❌ 受限于平台 |
| **成本（小规模）| ¥400-2,500/月 | ¥0-420/月 |
| **成本（中规模）| ¥1,600-6,000/月 | ¥1,140/月 |
| **运维要求** | 需要（Docker知识）| 无 |
| **可用性** | 自己保障 | Dify保障 |
| **客户交付** | ✅ 私有化优势 | ❌ 无法私有化 |
| **功能完整度** | ✅ 全部功能 | 部分限制 |
| **扩展性** | ✅ 无限 | 受套餐限制 |
| **模型选择** | ✅ 任意 | 受套餐限制 |

### 13.2 结论：**服务交付必须用自托管**

**理由**：
1. **客户要求数据私有化**：90%企业客户要求部署在自己服务器
2. **利润空间**：自托管可控成本，利润率更高
3. **差异化竞争力**：私有部署是核心卖点
4. **长期可控**：不依赖Dify平台政策变化

**Dify Cloud 适用场景**：
- 个人学习/实验
- 不要求私有化的小客户
- 快速原型验证（PoC阶段）

### 13.3 服务交付最佳实践

```
PoC阶段 → 用 Dify Cloud（免费版快速验证）
    ↓ 客户认可
正式项目 → 自托管部署到客户/你的服务器
    ↓ 
运维期 → 你负责托管（月费模式）
         或 客户自行运维（一次性交付）
```
