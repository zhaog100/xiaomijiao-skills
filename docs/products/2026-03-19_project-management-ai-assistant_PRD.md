## 1. 产品概述

### 1.1 产品名称与愿景

**ProjectMind（项目智脑）** — 一款AI原生的轻量级项目管理助手。

**愿景：** 让每位项目经理都拥有一位懂PMP方法论、会自动跟进项目、能主动预警风险的AI搭档。

**Slogan：** "懂项目的AI，不只是记任务。"

### 1.2 目标用户

| 用户群体 | 规模 | 痛点 |
|----------|------|------|
| 独立开发者/小团队 | 1-3人 | 不想学重型工具，需要快速上手 |
| 中小团队PM | 5-20人 | Jira太重，Notion太泛，需要专业PM功能 |
| 自由职业项目经理 | 个人 | 管理多个客户项目，需要高效报告和自动化 |

### 1.3 核心价值主张

1. **AI原生** — 用自然语言管理项目，不是填表单
2. **PMP方法论内置** — 基于PMBOK 7th原则设计，不是玩具
3. **轻量且专业** — 5分钟上手，但功能不缩水
4. **中文友好** — 原生中文体验，符合国内项目实践
5. **开放可扩展** — OpenClaw Skill + n8n/Dify 集成，不是封闭生态

### 1.4 与竞品差异化

| 维度 | Jira | Linear | Notion Projects | **ProjectMind** |
|------|------|--------|-----------------|-----------------|
| 上手成本 | 高（配置复杂） | 中 | 低 | **极低（AI对话即可）** |
| AI能力 | 插件式（弱） | 基础 | AI书写 | **AI原生（全流程）** |
| PMP支持 | 需定制 | 无 | 无 | **内置PMBOK 7th** |
| 中文友好 | 一般 | 差 | 一般 | **原生中文** |
| 部署方式 | 云/SaaS | SaaS | SaaS | **SaaS + 私有部署** |
| 价格（团队5人） | ~¥350/月 | ~¥175/月 | ~¥112/月 | ~¥210/月 | **¥145/月** |

## 3. 核心功能模块

### 3.1 MVP（v1.0）— 4周开发

> **策略调整（v1.1）：** MVP聚焦2个核心模块（智能任务管理 + 项目进度追踪），Skill版优先于Web版，会议纪要移至Phase 2。强调ClawHub作为第一分发渠道。

#### 模块一：智能任务管理

**用户故事：**
- **US-1.1** As a 独立开发者, I want 用自然语言创建任务, so that 我不用学习复杂的表单操作就能开始管理项目。
- **US-1.2** As a PM, I want AI自动将大需求拆解为子任务, so that 我不用花30分钟手动拆分，且拆解逻辑符合WBS最佳实践。
- **US-1.3** As a 团队成员, I want AI根据依赖关系和优先级自动排列任务顺序, so that 我知道下一步该做什么。
- **US-1.4** As a PM, I want 用中文自然语言修改任务状态（"把登录模块标记为完成"）, so that 我在手机上也能快速操作。

**功能点：**

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 自然语言任务创建 | 输入"开发用户注册功能，预计3天"自动创建任务 | P0 |
| AI任务拆解 | 输入"做一个电商系统"自动拆解为模块和子任务 | P0 |
| 智能优先级排序 | 基于MoSCoW方法自动标注优先级 | P1 |
| 任务依赖管理 | 自动识别任务间依赖关系，可视化依赖链 | P1 |
| 标签与分类 | 自动根据任务内容推荐标签 | P2 |

**AI安全机制（v1.1新增）：**

| 机制 | 描述 |
|------|------|
| 三步确认流程 | AI建议 → 用户确认（明确展示拆解结果） → 执行创建 |
| 置信度标注 | 每次AI拆解结果附带置信度评分（高/中/低），低置信度时提示用户复核 |
| WBS逻辑校验 | 拆解后自动验证：子任务是否覆盖父任务范围、是否有遗漏或冗余、粒度是否合理 |
| 幻觉防护 | 任务名称和描述中标注"AI生成，请核实"，关键数据（工时估算）默认需要确认 |

**验收标准：**
- [ ] 自然语言创建任务准确率 ≥ 90%
- [ ] AI拆解结果平均合理度（人工评估）≥ 80%
- [ ] 单次操作响应时间 < 3秒
- [ ] AI建议确认环节覆盖率 100%（所有AI生成的任务均需用户确认）
- [ ] WBS校验覆盖率 100%（拆解结果均通过逻辑校验）

#### 模块二：项目进度追踪

**用户故事：**
- **US-2.1** As a PM, I want 同时查看看板、甘特图和燃尽图, so that 我能从不同视角掌握项目全貌。
- **US-2.2** As a 团队Leader, I want 燃尽图自动更新, so that 我能实时判断Sprint是否健康。
- **US-2.3** As a 独立开发者, I want 看板视图拖拽即可更新状态, so that 我的管理成本趋近于零。

**功能点：**

| 视图 | 描述 | 优先级 |
|------|------|--------|
| 看板视图 | Todo/Doing/Review/Done 四列，拖拽操作 | P0 |
| 甘特图 | 时间线视图，显示任务起止时间和里程碑 | P1 |
| 燃尽图 | 按Sprint自动生成，显示理想vs实际曲线 | P1 |
| 进度百分比 | AI根据子任务完成度自动计算父任务进度 | P0 |

**AI安全机制（v1.1新增）：**

| 机制 | 描述 |
|------|------|
| 进度计算可解释 | AI自动计算的进度百分比附带计算依据（已完成X/Y子任务），用户可点击查看明细 |
| 异常标记确认 | AI检测到进度异常（如突然大跨度变化）时，标记待确认，不自动更新 |

**验收标准：**
- [ ] 看板拖拽流畅（60fps），状态变更实时同步
- [ ] 燃尽图数据延迟 < 30秒
- [ ] 甘特图支持 ≥ 100个任务不卡顿
- [ ] 进度计算依据展示覆盖率 100%

#### 模块三：每日站会摘要

> **v1.1调整：** 原模块三"AI会议纪要"移至Phase 2，MVP保留每日站会摘要作为轻量替代。

**用户故事：**
- **US-3.1** As a PM, I want AI自动收集团队成员的进展更新, so that 我不需要每天花15分钟开会。
- **US-3.2** As a 远程团队Lead, I want AI识别进展中的阻塞项, so that 我能主动协调解决。
- **US-3.3** As a 团队成员, I want 每天早上收到个人待办摘要, so that 我一天开始就知道该做什么。

**功能点：**

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 异步站会收集 | 团队成员随时提交"昨天做了/今天要做/阻塞项" | P0 |
| AI摘要生成 | 自动汇总生成站会纪要，标记阻塞项 | P0 |
| 定时推送 | 每日9:00推送站会摘要到QQ/Slack/企业微信 | P1 |
| 阻塞项追踪 | 自动跟踪阻塞项解决状态，超过2天未解决自动升级 | P1 |

**AI安全机制（v1.1新增）：**

| 机制 | 描述 |
|------|------|
| 摘要确认 | AI生成的站会摘要发布前需PM确认，防止遗漏关键信息 |
| 阻塞项置信度 | AI识别的阻塞项标注置信度，低置信度标注"建议人工核实" |

**验收标准：**
- [ ] 摘要覆盖所有团队成员提交内容（100%）
- [ ] 阻塞项识别准确率 ≥ 80%
- [ ] 定时推送准时率 ≥ 99%
- [ ] 摘要发布前确认率 100%

| 纪要模板 | 支持Sprint回顾、需求评审、每日站会等模板 | P2 |

**AI安全机制（v1.1新增）：**

| 机制 | 描述 |
|------|------|

| Action Item验证 | 提取的Action Items标注置信度，低置信度项需人工复核后再创建任务 |
| 转录完整性检查 | 自动检测语音转文字的遗漏段落（如静音过长、识别失败），提示人工补全 |

**验收标准：**
- [ ] 中文语音识别准确率 ≥ 95%
- [ ] Action Item提取准确率 ≥ 85%
- [ ] 从上传到生成纪要 < 2分钟（30分钟录音）
- [ ] 纪要发布前确认率 100%

#### 模块五：风险预警

**用户故事：**
- **US-5.1** As a PM, I want AI在进度偏差超过阈值时自动预警, so that 我能提前干预而不是事后补救。
- **US-5.2** As a PM, I want AI检测资源冲突（同一人被分配多个并行关键任务）, so that 我能及时调整资源分配。
- **US-5.3** As a 团队Lead, I want 风险仪表盘展示所有项目的风险状态, so that 一目了然知道哪个项目需要关注。

**功能点：**

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 进度偏差预警 | 计划vs实际偏差 > 15% 时标橙，> 30% 时标红 | P0 |
| 资源冲突检测 | 同一人同日被分配 > 8小时工作量时预警 | P0 |
| 风险登记册 | AI辅助识别风险，记录到风险登记册 | P1 |
| 里程碑延期预测 | 基于当前趋势预测里程碑是否按时交付 | P1 |

**数据质量评估（v1.1新增）：**

| 功能 | 描述 |
|------|------|
| 数据质量评分 | 对预警使用的数据源评估质量（1-5分），低质量数据降低预警置信度 |
| 数据来源可信度 | 标注数据来源（用户手动录入/系统自动采集/AI推断），可信度不同权重不同 |
| 数据完整性检测 | 检测任务缺少工时/截止日期等关键字段时，降低依赖该数据的预警等级 |
| 置信度联动 | 数据质量低时，预警从"高危"降级为"待确认"，避免误报 |

**AI安全机制（v1.1新增）：**

| 机制 | 描述 |
|------|------|
| 预警确认 | 高影响预警（如里程碑延期）需PM确认后才发送通知 |
| 预警置信度 | 每条预警附带置信度百分比，结合数据质量评分动态调整 |
| 误报学习 | 用户标记"误报"后，AI自动调整该类预警的阈值和权重 |

**验收标准：**
- [ ] 进度偏差检测准确率 ≥ 90%
- [ ] 资源冲突检测覆盖率 100%
- [ ] 预警通知延迟 < 5分钟

### 4.1 整体架构

```
┌─────────────────────────────────────────────┐
│                   客户端层                    │
│  Next.js Web App │ QQ Bot │ Slack │ CLI     │
├─────────────────────────────────────────────┤
│                   API网关层                   │
│  认证鉴权 │ 速率限制 │ 路由 │ WebSocket     │
├─────────────────────────────────────────────┤
│                   业务逻辑层                   │
│  任务管理 │ 进度追踪 │ 会议纪要 │ 报告生成      │
├─────────────────────────────────────────────┤
│                   AI服务层                    │
│  LLM编排 │ 语音识别 │ 向量检索 │ Prompt管理   │
│  输出校验 │ 人工确认 │ 置信度评估 │ 幻觉检测     │
├─────────────────────────────────────────────┤
│                   数据存储层                   │
│  PostgreSQL │ Redis │ SQLite(嵌入式) │ S3    │
└─────────────────────────────────────────────┘
```

### 4.2 技术选型

| 层级 | 技术 | 理由 |
|------|------|------|
| **前端** | Next.js 14 + TailwindCSS | SSR/SSG灵活、生态成熟、部署方便 |
| **后端** | Node.js (Hono/Fastify) | 轻量高性能、与前端技术栈统一 |
| **AI层** | OpenAI API + Ollama（本地） | 云端保证质量、本地保证隐私和成本 |
| **数据库** | PostgreSQL（SaaS）/ SQLite（CLI版） | 生产级可靠性 / 零配置嵌入 |
| **缓存** | Redis | 会话管理、实时通知、速率限制 |
| **消息队列** | BullMQ（基于Redis） | 异步任务处理（AI推理、报告生成） |
| **文件存储** | S3兼容（MinIO自托管/云） | 录音文件、导出报告存储 |
| **部署** | Docker Compose + Vercel | 一键部署、弹性伸缩 |
| **监控** | OpenTelemetry + Grafana | 全链路可观测 |

### 10.6 HN趋势洞察（2026-03-20新增）

> 以下整理自 Hacker News 热门文章（1355-148 pts），作为产品差异化和技术选型参考。

| 趋势 | 来源 | 对PRD的影响 |
|------|------|------------|
| **Astral(Ruff/uv)被OpenAI收购** — Python工具链+AI编码深度融合，开源承诺不变 | HN 1355🔥 | 验证了"AI+开发工具"赛道的价值；ProjectMind应强化AI原生定位而非传统PM工具 |
| **KittenTTS <25MB** — 15M-80M参数ONNX模型，纯CPU推理，8种语音 | HN 422🔥 | 可集成语音功能：语音创建任务、语音查看进度、语音播报周报，降低使用门槛 |
| **Claude Channels** — 推送事件到后台运行的Agent，Agent自动响应（Telegram/Discord） | HN 330🔥 | 验证了OpenClaw "Agent常驻+消息推送"的方向；ProjectMind Skill版可复用此交互模式 |
| **Autoresearch × 16GPU** — Agent自动做910个实验，9x加速，学会"廉价筛选+高价验证"策略 | HN 174🔥 | AI自主实验+自动优化的模式可借鉴：ProjectMind可自动分析项目数据趋势，给出优化建议 |
| **NanoGPT Slowrun 10x数据效率** — 18B参数集成(1.8B×10)，用100M tokens达到1B效果 | HN 148🔥 | 集成学习思想可应用于模型路由：多个轻量模型投票决策，比单个大模型更稳定 |

**对ProjectMind的核心启示：**

1. **AI+工具融合不可逆** — 不是"加个AI功能"，而是"AI即产品"。ProjectMind每个交互都应是AI驱动的。
2. **轻量化是王道** — 25MB TTS能替代GB级模型，说明PM工具不需要重型部署。Skill版优先路线正确。
3. **Agent常驻交互** — Claude Channels证实了"后台Agent+推送"模式。ProjectMind Skill版应支持主动提醒和定时报告。
4. **自动优化** — 不只是展示数据，而是自动分析趋势并给出建议。这是"智脑"的核心差异化。
