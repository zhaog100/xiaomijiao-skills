## 1. 需求概述

### 1.1 背景

**问题**：
- 小米辣（PM 代理）和小米粒（Dev 代理）需要高效协作
- 飞书 Bot 机制限制：Bot 发送的消息不会推送给其他 Bot
- 之前的技术方案（独立双 Bot、共享服务）全部失败

**根本原因**：
- 飞书开放平台默认不推送 Bot 消息给其他 Bot
- 需要申请企业自建应用权限（im:message.group_msg）

### 1.2 目标

**核心目标**：实现双米粒在飞书群的实时协作沟通

**具体目标**：
1. ✅ 小米辣和小米粒能互相看到对方消息
2. ✅ 支持@提及和路由
3. ✅ 消息不丢失（100% 可靠）
4. ✅ 延迟 < 5 秒

### 1.3 范围

**包含**：
- 企业自建应用权限申请
- 消息分发器开发
- 双 Bot 处理器
- 共享数据库
- 消息路由逻辑

**不包含**（后续版本）：
- 其他平台集成（企业微信/钉钉）
- 复杂对话流程编排
- 多群管理

## 4. 技术架构

### 4.1 整体架构

```
┌─────────────────────────────────────────┐
│         飞书开放平台                     │
│  (企业自建应用 + im:message.group_msg)   │
└─────────────┬───────────────────────────┘
              │ 读取所有群消息
              ↓
┌─────────────────────────────────────────┐
│         消息分发器                       │
│  (判断消息来源，分发给对应 Bot)           │
└─────────────┬───────────────────────────┘
              ↓              ↓
       小米辣处理器    小米粒处理器
              ↓              ↓
       存储 + 回复      存储 + 回复
              ↓              ↓
       调用飞书 API 代发消息
              ↓
       飞书群显示（双方都能看到）
```

### 4.2 技术栈

| 组件 | 技术选型 | 说明 |
|------|---------|------|
| 后端框架 | Python 3.12 + FastAPI | 异步高性能 |
| 飞书 SDK | lark-oapi v1.5.3 | 官方 SDK |
| 数据库 | SQLite | 轻量级本地存储 |
| 消息队列 | Redis Pub/Sub | 可选，用于解耦 |
| 部署 | Systemd | 自动启动 + 崩溃重启 |

## 5. 数据库设计

### 5.1 消息表（messages）

```sql
CREATE TABLE messages (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    message_id TEXT NOT NULL,          -- 飞书消息 ID
    sender_id TEXT NOT NULL,           -- 发送者 ID
    sender_type TEXT NOT NULL,         -- 发送者类型（user/bot）
    content TEXT NOT NULL,             -- 消息内容
    chat_id TEXT NOT NULL,             -- 群聊 ID
    timestamp DATETIME DEFAULT CURRENT_TIMESTAMP,
    message_type TEXT DEFAULT 'text',  -- 消息类型
    is_read BOOLEAN DEFAULT FALSE,     -- 是否已读
    archived BOOLEAN DEFAULT FALSE     -- 是否归档
);
```

### 5.2 Bot 状态表（bot_status）

```sql
CREATE TABLE bot_status (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    bot_name TEXT NOT NULL,            -- Bot 名称
    status TEXT NOT NULL,              -- 状态（running/stopped）
    last_heartbeat DATETIME,           -- 最后心跳时间
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

## 6. API 接口定义

### 6.1 飞书 API

| 接口 | 方法 | 用途 |
|------|------|------|
| /open-apis/im/v1/messages | POST | 发送消息 |
| /open-apis/auth/v3/tenant_access_token/internal | POST | 获取令牌 |

### 6.2 内部 API

| 接口 | 方法 | 用途 |
|------|------|------|
| /api/v1/messages | GET | 查询消息列表 |
| /api/v1/messages/{id} | GET | 查询单条消息 |
| /api/v1/bot/status | GET | 查询 Bot 状态 |

## 9. 风险评估

### 9.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 权限审核不通过 | 中 | 高 | 准备备选方案（官家中转） |
| 消息延迟过高 | 低 | 中 | 优化代码，使用 Redis |
| 服务崩溃 | 低 | 高 | Systemd 自动重启 |

### 9.2 业务风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| 飞书 API 限流 | 低 | 中 | 增加重试机制 |
