## 1. 产品概述

### 1.1 灵感来源

HN热门项目 [Lucidic AI](https://dashboard.lucidic.ai)（Stanford团队）展示了Agent调试的巨大需求：
- Agent运行轨迹可视化
- "时间旅行"：修改任意状态后重新模拟
- 轨迹聚类：自动分组相似运行路径
- 自定义eval（rubrics）

但Lucidic需要手动接入（`lai.init()`），目标是通用AI Agent。

### 1.2 我们的差异化

**AgentLens 专为 OpenClaw 设计** — 零接入成本，OpenClaw Agent的每次对话本身就是调试数据。

| 维度 | Lucidic | LangSmith | **AgentLens** |
|------|---------|-----------|---------------|
| 接入方式 | 手动SDK | 手动SDK | **零接入（自动采集）** |
| 数据来源 | Agent日志 | Agent日志 | **OpenClaw原生日志+会话数据** |
| 目标用户 | AI研究员 | 开发团队 | **AI Agent开发者/使用者** |
| 部署 | 云SaaS | 云SaaS | **本地/私有（Skill版）** |
| 价格 | 免费额度+付费 | $50+/月 | **免费+ClawHub** |

### 1.3 核心价值主张

1. **零接入** — 安装Skill即用，无需改代码
2. **全链路追踪** — 用户消息→Agent解析→Tool调用→响应，完整链路
3. **浪费检测** — 复用ai-efficiency-monitor的5种模式
4. **失败分析** — 自动聚类失败路径，找到根因
5. **成本优化** — Token消耗可视化+优化建议

## 2. 目标用户

### 用户画像一：AI Agent开发者（官家本人）
- **场景**：开发OpenClaw技能，需要调试子代理行为
- **痛点**：子代理经常超时、commit未保存、互相改坏文件，但缺乏可视化追踪
- **需求**：看到子代理的完整执行链路，快速定位失败节点

### 用户画像二：AI Agent使用者
- **场景**：使用OpenClaw管理日常任务
- **痛点**：不知道Agent在哪里浪费时间、哪些调用是无效的
- **需求**：效率报告+成本优化建议

### 用户画像三：OpenClaw生态开发者
- **场景**：为OpenClaw开发技能或插件，需要调试运行时行为
- **痛点**：缺乏官方调试工具，只能看日志
- **需求**：开源轻量、零配置的Agent行为追踪工具

## 3. MVP功能模块（2周）

### 模块一：会话追踪器（Session Tracer）

**核心功能：**
- 解析OpenClaw会话历史，提取完整执行链路
- 每个步骤标注：时间/耗时/Token数/Tool名称/状态(成功/失败/超时)
- 生成ASCII时序图

**示例输出：**
```
[18:00:01] 👤 用户: "帮我创建一个任务"
[18:00:02] 🤖 Agent解析意图 (320 tokens, 1.2s) ✅
[18:00:03] 🔧 pm:create-task (180 tokens, 0.8s) ✅
[18:00:03] 📝 响应格式化 (50 tokens, 0.1s) ✅
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
总耗时: 2.1s | 总Token: 550 | 成功率: 100%
```

### 模块二：失败分析器（Failure Analyzer）

**核心功能：**
- 自动识别失败的Tool调用
- 按失败类型聚类：超时/错误/参数缺失/依赖缺失
- 生成失败路径树

**示例：**
```
📊 失败分析（近7天）
━━━━━━━━━━━━━━━━━━
总执行: 47次 | 失败: 12次 (25.5%)

失败分布:
  ⏱️ 超时 (5) ████████ 41.7%
  ❌ 错误 (4) ██████   33.3%
  ⚠️ 参数缺失 (3) ████   25.0%

最常失败: sessions_spawn → 子代理超时
建议: 将复杂任务拆分为<3分钟的子任务
```

### 模块三：效率仪表盘（Efficiency Dashboard）

**复用ai-efficiency-monitor已有能力：**
- Token消耗趋势（日/周/月）
- 浪费模式检测（5种）
- 模型成本计算
- ASCII趋势图

### 模块四：子代理监控（Sub-agent Monitor）

**核心功能：**
- 追踪子代理的启动/完成/超时状态
- 并行子代理的资源使用情况
- 子代理间冲突检测（文件修改冲突）

## 4. 技术架构

### 4.1 数据采集层（双源策略）

数据可行性是AgentLens最大的技术风险。采用**双源策略**确保数据可用性：

| 数据源 | 优先级 | 获取方式 | 数据内容 | 限制 |
|--------|--------|----------|----------|------|
| **OpenClaw API** | 主 | `sessions_history` + `sessions_list` | 会话消息（含tool_calls元数据）、子代理状态 | 返回消息文本，tool调用详情可能不完整 |
| **OpenClaw日志文件** | 备 | 解析 `~/.openclaw/logs/` | 结构化日志（timestamp、token数、API调用、错误） | 需要文件系统访问权限 |

**Day 1 Spike验证（30分钟）：**
1. 调用 `sessions_history` 查看实际返回数据结构
2. 检查日志文件格式和字段
3. 根据结果决定主数据源

**降级方案：**
- 如果API数据不足 → 以日志文件为主
- 如果两者都不够 → 仅支持"手动标注"模式（用户提供执行步骤，AgentLens做分析）

### 4.2 技术选型

| 组件 | 技术 | 理由 |
|------|------|------|
| 数据存储 | SQLite | 零配置，与ProjectMind统一 |
| 数据解析 | Bash + Node.js | 复用ai-efficiency-monitor的bash脚本 |
| 可视化 | ASCII/Markdown | Skill版无需GUI，OpenClaw原生展示 |
| 会话查询 | OpenClaw sessions_history API | 直接调用 |

## 6. 商业模式

### 策略：开源获客 → 生态集成 → 定制服务

**MVP阶段完全免费开源**，不设付费墙。积累用户和GitHub Star后通过以下路径商业化：

| 阶段 | 时间 | 策略 | 目标 |
|------|------|------|------|
| **种子期** | 第1-3月 | GitHub开源+ClawHub分发 | ≥200 Star，≥100安装 |
| **成长期** | 第4-6月 | 争取OpenClaw官方集成 | 成为核心调试工具 |
| **商业化** | 第6月+ | 技能咨询/定制服务 | 按项目收费 ¥2,000-10,000 |

> 不卖订阅。OpenClaw用户群太小，SaaS模式不成立。
> 价值在于：开源影响力 → 技能咨询需求 → 官方生态位。

## 9. AI安全机制

| 机制 | 说明 |
|------|------|
| **置信度标注** | 失败分析结果附带置信度（0-1），低置信度标注"建议核实" |
| **幻觉防护** | AI生成的建议/根因分析标记`ai_generated=true`，描述追加"[AI分析，请核实]" |
| **用户确认** | 重大建议（如"删除大量数据"）需用户确认后执行 |
| **数据本地化** | 所有分析数据仅存本地SQLite，不外传 |
| **无自动操作** | AgentLens只做分析和报告，不自动修改任何文件或配置 |
