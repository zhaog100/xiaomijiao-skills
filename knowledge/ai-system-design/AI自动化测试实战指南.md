# AI自动化测试实战指南

_从踩坑到成功的完整方法论_

---

## 一、核心思想：先规划，再执行

### 1.1 关键认知转变

**错误做法**：
- 直接让AI测试全站
- 截图引导测试
- 简单的循环测试

**正确做法**：
1. AI学习网站功能
2. 根据大纲生成测试用例
3. 准备测试数据
4. 执行测试并记录

### 1.2 类比理解

**之前的做法**：
> 领导跟实习生说"你去把那个项目测一下"，然后什么背景都不介绍，什么要求都不说。结果：实习生一脸懵逼，随便点点点就算测完了。

**现在的做法**：
> 领导先跟实习生说"这是项目的背景，这是要测的功能，这是测试的步骤，这是要准备的数据，你看明白了再动手"。结果：测得又快又好，还不漏东西。

## 二、工具配置：Playwright MCP

### 2.1 安装配置

在 Claude Code 的 MCP 配置文件中添加：
```json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["-y", "@playwright/mcp@latest"]
    }
  }
}
```

重启 Claude Code，配置完成。

### 2.2 核心能力

- 打开浏览器，自动化操作
- 截图记录
- 执行测试步骤
- 生成测试报告

## 三、探索过程：从失败到成功

### 3.1 第一次尝试：直接测试全站

**方法**：告诉AI"把整个网站的功能都测试一遍"

**结果**：
- AI打开页面，测了几个页面就停了
- 容易遗漏功能
- 不知道要测什么

**教训**：直接让AI测试，它缺乏目标和方向

### 3.2 第二次尝试：截图引导测试

**方法**：截了个图，让AI按着截图上的功能一个个测试

**结果**：
- 比之前好一些
- 但还是有些功能没覆盖到
- 还是不够系统

**教训**：图片引导有改进，但不够完整

### 3.3 第三次尝试：循环测试每个模块

**方法**：让AI打开页面后，对每个模块循环测试

**结果**：
- 每个模块都点到了
- 但深层功能没有覆盖到（如话术编辑器、维度管理、规则配置）

**教训**：测试不能只是点点页面，需要更系统的方法

### 3.4 持续探索

尝试方案：
- 让AI自己学习页面结构，然后写测试脚本
- 把测试步骤写得超级详细
- 分模块让AI测试

**瓶颈**：总觉得还不够完美，缺少更高效的方法

### 3.5 转机：从subAgent得到的灵感

**灵感来源**：subAgent分析商业模式的流程
1. 先让AI去抓各种资料回来汇总
2. 然后列一个分析大纲
3. 用户确认大纲后，AI才开始分析
4. 最后给出完整的分析报告

**关键洞察**：为什么不让AI先学习一下网站，再测试呢？

## 四、最终方案：四步走

### 4.1 第一步：AI学习网站功能

**操作**：
1. 让AI打开浏览器，把整个网站的功能都过一遍
2. 罗列出所有功能模块
3. 第一次罗列可能不全，再让AI打开看看，补充遗漏的功能
4. 经过几次补充，整站的功能大纲就出来了

**示例输出**（某医美智能工牌系统）：
- 登录模块（账号登录、短信登录、忘记密码）
- 管理驾驶舱
- 客户中心（来访记录、客户档案）
- 运营中心（模型管理、到院提示词）
- 录音管理
- 接诊填报
- 运营中台（客户白描、接诊评价、客户关注）
- 系统设置（偏好设置、热词管理）

### 4.2 第二步：根据大纲生成测试用例

**操作**：针对每个功能点写测试用例

**示例输出**（登录功能）：
- 正常登录成功
- 账号错误
- 密码错误
- 手机号为空
- 密码为空
- 手机号格式错误
- 短信验证码登录
- 验证码错误
- 验证码过期
- 重复获取验证码
- 忘记密码流程

**成果**：11个场景用例，覆盖各种场景

### 4.3 第三步：准备测试数据

**操作**：为测试准备必要的数据

**示例**：
- 测试客户搜索功能，需要准备客户姓名、手机号、会员号等测试数据

### 4.4 第四步：执行测试并记录

**操作**：
1. AI按照测试用例一条条执行
2. 每个测试异常都截图
3. 测试完成后生成报告

**成果**：
- 154个测试用例，3个小时跑完
- 每个测试都有截图证据
- 测试报告详细记录了哪个通过、哪个失败、失败原因是什么

## 六、效率对比

### 6.1 传统手动测试

- **时间**：几天
- **风险**：人可能会漏测
- **重复性**：每次改完代码需要重新手动测试

### 6.2 AI自动化测试

- **时间**：3小时
- **优势**：AI不会漏测，每次改完代码都能快速回归测试
- **证据**：每个测试都有截图证据

**效率提升**：**几天 → 3小时**

## 七、延伸思考：AI与人的分工

### 7.1 什么样的测试会被替代？

- 重复性高的回归测试
- 逻辑清晰的功能测试
- 大量的表单填写和数据验证
- 需要频繁执行的冒烟测试

### 7.2 什么样的测试需要人？

- 复杂的业务逻辑分析
- 测试策略和测试计划设计
- 探索性测试和用户体验评估
- 需要业务理解和场景判断的测试

### 7.3 程序员怎么用好AI？

**核心思路**：把AI当成超级助手，而不是替代品

**具体分工**：
- 别让AI直接干活，先让它规划和学习
- 你负责设计思路，让AI负责执行
- 你负责业务判断，让AI负责重复操作
- 你负责解决复杂问题，让AI负责常规任务

**时间分配**：
- 程序员花时间最多的是：搞定业务、设计测试用例、分析测试结果
- AI帮忙做的是：打开浏览器、执行测试步骤、记录结果

**理念**：程序员的时间应该花在思考和设计上，而不是重复执行上

### 7.4 普通人怎么用好AI？

**核心原则**：你做"大脑"，AI做"手脚"

**应用场景**：
1. **写文案**：你负责创意和框架，AI负责扩充和润色
2. **做方案**：你负责目标和策略，AI负责细节和执行
3. **学技能**：你负责规划学习路径，AI负责解答和练习
4. **做决策**：你负责权衡利弊，AI负责提供信息和数据

**核心思想**：
- 你负责想清楚要做什么、为什么做、做到什么程度
- AI负责查资料、整理信息、生成内容、执行任务

## 八、方法论总结

### 8.1 核心原则

**先规划，再执行**

就像软件开发一样：
- 熟悉需求
- 设计架构
- 技术设计
- 编写代码
- 测试调试

AI测试也是一样的道理：**先规划，再执行**

### 8.2 规划四步法

1. **学习网站功能**（AI打开浏览器，罗列出所有功能模块）
2. **生成测试用例**（根据功能大纲，写出详细的测试用例）
3. **准备测试数据**（准备测试需要的账号、姓名、手机号等数据）
4. **执行测试并记录**（按用例执行，截图，生成报告）

### 8.3 关键认知

**把AI当实习生**：
- 它不知道要做什么，你得教它
- 它需要背景、目标、步骤、数据
- 教好了，它能做得又快又好
- 不教，它就随便做做

**别把AI当神仙**：
- 它就像个新来的实习生
- 你得教它怎么做，它才能干好

## 九、未来展望

### 9.1 AI不会完全代替测试人员

AI会改变测试工作的方式，而不是完全替代。

### 9.2 人的价值

- 复杂的业务逻辑分析
- 测试策略设计
- 探索性测试
- 用户体验评估

### 9.3 AI的价值

- 快速执行重复性任务
- 不遗漏任何测试点
- 提供详细的测试证据
- 支持快速回归测试

### 9.4 协作模式

**人机协作**：
- 人做思考和设计
- AI做执行和记录
- 人做业务判断
- AI做重复操作
