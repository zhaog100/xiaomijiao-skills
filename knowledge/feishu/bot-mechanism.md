# 飞书Bot机制详解

## 🚨 核心限制

### Bot消息不推送给其他Bot

**机制**：
```
Bot发送的消息 → 不会推送给其他Bot
```

**设计原因**：防止Bot循环推送

**影响**：
- ❌ Bot↔Bot无法直接沟通
- ❌ 无法实现自动协作
- ✅ 必须通过用户中转或申请权限

---

## 📋 飞书Bot类型

### 1. 自定义机器人（Webhook）

**用途**：Bot主动发送消息到群

**特点**：
- ✅ 简单易用
- ✅ 无需审核
- ❌ 单向推送（Bot→群）
- ❌ 无法接收消息

**适用场景**：
- 定时推送
- 告警通知
- 单向广播

---

### 2. 企业自建应用（Bot）

**用途**：接收和发送消息

**特点**：
- ✅ 双向通信
- ✅ 功能完整
- ⏳ 需要审核
- ⚠️ 默认只推@消息

**权限申请**：
```
im:message.group_msg - 获取群组消息（不@也能收到）
```

**适用场景**：
- Bot客服
- 智能助手
- 群管理

---

## 🔍 消息推送规则

### 自定义机器人
```
Bot → 群：✅ 可以
群 → Bot：❌ 不可以
Bot ↔ Bot：❌ 不可以
```

### 企业自建应用（默认）
```
用户@Bot → Bot：✅ 可以
用户不@ → Bot：❌ 不可以
Bot → 群：✅ 可以
Bot ↔ Bot：❌ 不可以
```

### 企业自建应用（申请权限后）
```
用户@Bot → Bot：✅ 可以
用户不@ → Bot：✅ 可以
Bot → 群：✅ 可以
Bot ↔ Bot：❌ 不可以
```

| 平台 | Bot入群 | Bot↔Bot | 限制 |
|------|---------|---------|------|

## 💡 解决方案

### 方案1：用户中转（立即可用）

**流程**：
```
用户 → Bot1：问题
Bot1 → 用户：回复
用户 → Bot2：Bot1说X
Bot2 → 用户：回复
```

**优点**：
- ✅ 立即可用
- ✅ 无需审核

**缺点**：
- ⚠️ 需要用户手动中转

---

### 方案2：企业自建应用权限（1-3天）

**流程**：
```
1. 申请权限：im:message.group_msg
2. 等待审核：1-3天
3. 审核通过 → 不@也能收到
```

**优点**：
- ✅ 官家省事
- ✅ 官方支持

**缺点**：
- ⏳ 需要等待审核

---

## 🔧 技术实现

### 事件订阅

**Request URL**：
```
http://your-server.com/webhook
```

**事件**：
```
im.message.receive_v1 - 接收消息
```

**推送内容**：
- 默认：只推@消息
- 申请权限后：推送所有消息

---

### Webhook

**配置**：
```
群设置 → 群机器人 → 添加自定义机器人
```

**用途**：
```
Bot主动发送消息到群
```

**限制**：
```
❌ 无法接收群消息
```

---

## 📝 最佳实践

### 1. 先验证可行性
```
✅ 先测试Bot能否互相看到
✅ 确认平台限制
✅ 设计符合规则的方案
```

### 2. 理解平台限制
```
✅ 飞书不推送Bot消息
✅ 需要权限才能接收所有消息
✅ Webhook是单向的
```

### 3. 选择正确方案
```
立即可用 → 用户手动@
1-3天后 → 申请权限
长期 → 使用支持Bot↔Bot的平台
```

## 🚨 常见错误

### 错误1：认为Bot能互相看到
```
❌ 假设：Bot发的消息，其他Bot能收到
✅ 真相：飞书不推送Bot消息给其他Bot
```

### 错误2：认为Webhook能接收消息
```
❌ 假设：Webhook是双向的
✅ 真相：Webhook是单向的（Bot→群）
```

### 错误3：试图绕过平台限制
```
❌ 假设：技术能绕过限制
✅ 真相：必须遵守平台规则
```
