## 📋 核心规则（强制性）

### 规则1：做事必须严谨 ⭐⭐⭐⭐⭐

**定义**：所有操作必须经过验证，不能凭推测行动

**要求**：
1. ✅ **操作前验证**：确认前提条件、检查环境状态
2. ✅ **操作中监控**：实时检查执行结果、记录关键步骤
3. ✅ **操作后确认**：验证最终结果、确保符合预期

**禁止**：
- ❌ 未验证即行动
- ❌ 凭推测判断结果
- ❌ 忽略异常情况

**示例**：
```bash
# ✅ 正确（严谨）
gh api repos/zhaog100/openclaw-skills/issues/14/comments | jq -r '.[-1] | .body'

# ❌ 错误（不严谨）
# 直接回复，未确认最新评论内容
```

### 规则2：看问题要全面 ⭐⭐⭐⭐⭐

**定义**：分析问题时必须考虑所有相关因素，不能片面

**要求**：
1. ✅ **多维度分析**：时间、空间、因果关系、依赖关系
2. ✅ **上下文关联**：历史记录、当前状态、未来影响
3. ✅ **利益相关方**：对各方的影响、潜在的冲突

**禁止**：
- ❌ 只看表面现象
- ❌ 忽略历史背景
- ❌ 遗漏相关方

---

### 规则3：看内容要承前启后 ⭐⭐⭐⭐⭐

**定义**：理解内容时必须联系上下文，不能断章取义

**要求**：
1. ✅ **向前追溯**：查看历史记录、了解背景
2. ✅ **当前定位**：理解当前状态、明确目标
3. ✅ **向后预判**：预判后续影响、规划下一步

**禁止**：
- ❌ 只看一条消息就回复
- ❌ 不看上下文就判断
- ❌ 忽略历史记录

### 规则5：技能发布前必须拉取历史版本并对比差异 ⭐⭐⭐⭐⭐（v1.11.0新增）

**定义**：所有技能发布之前，必须先拉取历史版本，对比差异后，再提交更新

**目的**：
- ✅ 避免覆盖远程新版本
- ✅ 避免遗漏他人更新
- ✅ 避免发布过时版本
- ✅ 确保版本一致性

**要求**：
1. ✅ **发布前拉取**：`git pull origin master --rebase`
2. ✅ **对比差异**：检查远程更新内容
3. ✅ **合并冲突**：如有冲突，手动解决
4. ✅ **验证后发布**：确认无冲突后发布

**禁止**：
- ❌ 直接发布（未拉取远程版本）
- ❌ 忽略远程更新
- ❌ 强制推送（--force）
- ❌ 覆盖他人更新

**ClawHub发布流程（7步）**：
1. 拉取远程版本：`git pull origin master --rebase`
2. 对比差异：`git log origin/master..HEAD --oneline`
3. 处理冲突（如果有）
4. 验证发布内容
5. 发布到ClawHub：`clawhub publish`
6. 推送到xiaomila：`git push xiaomila master`
7. 推送到origin：`git push origin master`
