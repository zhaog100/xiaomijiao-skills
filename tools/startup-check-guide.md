# 启动上下文检测指南

## 🎯 目标
在会话启动后立即检测上下文占用，确保启动内容控制在合理范围内。

## 📋 检测流程

**步骤1**：会话启动后，立即调用
```bash
session_status
```

**步骤2**：检查"Context:"行的百分比

**步骤3**：根据占用率给出建议

## 📊 判断标准

| 占用率 | 状态 | 建议 |
|--------|------|------|
| <10% | ✅ 优秀 | 启动内容精简，剩余空间充足 |
| 10-20% | ✅ 良好 | 启动内容合理，可以继续 |
| 20-30% | ⚠️ 注意 | 建议检查是否读取了不必要的文件 |
| 30-50% | ⚠️ 警告 | 建议精简启动内容，使用分层读取 |
| >50% | 🚨 严重 | 立即优化启动流程！ |

## 🔧 优化措施

**如果启动占用 >30%**：

1. **检查启动读取的文件**：
   - ❌ 是否读取了MEMORY.md（完整版，30KB+）？
   - ❌ 是否读取了历史日志（memory/YYYY-MM-DD.md，70KB+/天）？
   - ❌ 是否读取了其他大文件？

2. **优化方案**：
   - ✅ 只读MEMORY-LITE.md（精简版，<10KB）
   - ✅ 历史日志改为QMD检索（按需读取）
   - ✅ 详细信息改为QMD检索（节省90% tokens）

3. **实施方法**：
   - 修改AGENTS.md启动流程
   - 使用分层读取策略
   - 启动后主动检测上下文

## 💡 最佳实践

**理想的启动流程**：
```
启动时只读：
├── 核心层（<5KB）
│   ├── SOUL.md（身份）
│   └── USER.md（用户信息）
│
├── 摘要层（<10KB）
│   └── MEMORY-LITE.md（核心记忆）
│
└── 详情层（按需，QMD检索）
    ├── MEMORY.md（完整记忆）
    └── memory/YYYY-MM-DD.md（历史日志）
```

**预期效果**：
- 启动占用：<10%
- 剩余空间：90%
- 数据不丢失：QMD精准检索

## 📝 记录要求

**如果检测到启动占用 >30%**：
1. 记录到memory/YYYY-MM-DD.md
2. 分析读取了哪些文件
3. 提供优化建议
4. 更新AGENTS.md（如果需要）
