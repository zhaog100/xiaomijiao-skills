## 🔒 沙箱隔离机制

**核心原则：**
- 工具在 Docker 容器中执行
- 子代理保持沙箱隔离（无提升/宿主访问）
- 需要沙箱外的读写 → **不要启动子代理，先询问**

**配置项：**
- 沙箱工作区：`{sandbox_workspace_dir}`
- 代理工作区访问：`{workspace_access}`（挂载于 `{mount_path}`）
- 沙箱浏览器：已启用
- 沙箱浏览器观察器（noVNC）：`{novnc_url}`
- 宿主浏览器控制：可配置（允许/阻止）

**提升权限级别：**
- `off` - 无提升权限
- `ask` - 在宿主上运行exec并需要审批
- `full` - 自动审批

## 😊 表情反应指导

### 最小模式（minimal）
**适用场景：** 需要克制的频道

**行为规范：**
- 仅在真正相关时反应
- 每5-10次交流最多1次反应
- 确认重要的用户请求
- 偶尔表达真实情感（幽默、感激）
- 避免对常规消息或自己的回复做反应

### 广泛模式（extensive）
**适用场景：** 可以自由表达的频道

**行为规范：**
- 自由地反应
- 用合适的表情确认消息
- 通过反应表达情感和个性
- 对有趣的内容、幽默或值得注意的事件做反应
- 用反应确认理解或同意

**QQ Bot当前状态：** 未启用（inline buttons不可用）

## 🧠 推理格式规则

**标准格式：**
```
<reasoning>
简短的内部推理
</reasoning>
<reply>
用户可见的回复
</reply>
```

**核心规则：**
1. 所有内部推理必须在 `<reasoning>` 内
2. 不要在 `<reasoning>` 外输出任何分析
3. 每次回复格式：`<reasoning>...</reasoning>` 然后 `<reply>...</reply>`
4. 只有 `<reply>` 内的文字会展示给用户
5. `<reasoning>` 外的内容会被丢弃，用户永远看不到

## 💡 核心洞察

### 1. 架构分层清晰
- **引擎层** - 硬编码，不可修改（system-prompt.ts生成）
- **配置层** - 文件可自定义（AGENTS/SOUL/TOOLS等）
- **运行时层** - 动态信息（时间、模型、channel等）

### 2. 加载顺序有逻辑
- 先加载行为规范（AGENTS）→ 明确"怎么做"
- 再加载人格定义（SOUL）→ 明确"我是谁"
- 然后加载工具备忘（TOOLS）→ 明确"用什么"
- 最后加载身份和用户（IDENTITY/USER）→ 明确"服务谁"

### 3. 沙箱安全重要
- 默认沙箱隔离，工具在Docker中执行
- 子代理保持隔离，无提升/宿主访问
- 需要外部访问时先询问，不要自行启动子代理

### 4. 反应要克制
- 最小模式：5-10次交流1次反应
- 广泛模式：随时可反应
- 避免过度反应，保持自然

### 5. 推理要内部化
- 用户看不到 `<reasoning>` 内容
- 只展示 `<reply>` 部分
- 思考过程透明可控

### 6. 子代理裁剪精准
- 仅保留核心工具和安全部分
- 省略记忆、身份、消息等
- 保持轻量、高效、隔离

## 🎯 实践应用

### 会话启动流程
```
1. 读取 AGENTS.md（行为规范）
2. 读取 SOUL.md（人格定义）
3. 读取 IDENTITY.md（身份标识）
4. 读取 USER.md（用户画像）
5. 读取 TOOLS.md（工具备忘）
6. 读取 memory/YYYY-MM-DD.md（今天+昨天）
7. 如果主会话 → 读取 MEMORY.md
```

### 心跳执行流程
```
1. 读取 HEARTBEAT.md
2. 检查1-2项（邮件/日历/社交/天气/项目）
3. 有价值才联系
4. 无重要事项 → HEARTBEAT_OK
```

### 子代理启动流程
```
1. 仅加载工具+安全+CLI+工作区+运行时
2. 省略其他所有章节
3. 保持沙箱隔离
4. 无提升/宿主访问
```
