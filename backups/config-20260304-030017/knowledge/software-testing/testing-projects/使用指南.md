## 🚀 快速开始

### 1. 创建新项目

**步骤**：
1. 复制 `PROJECT-TEMPLATE.md` 到新项目目录
```bash
cd knowledge/testing-projects/
mkdir [项目名称]
cp PROJECT-TEMPLATE.md [项目名称]/README.md
```

2. 修改项目README
   - 填写项目基本信息
   - 定义测试范围
   - 制定测试计划

3. 创建项目子目录
```bash
cd [项目名称]/
mkdir test-cases test-reports test-plans test-scripts assets
```

4. 在 `PROJECT-INDEX.md` 中添加项目索引

### 2. 编写测试用例

**模板结构**：
```markdown
# [模块名] - 测试用例

## TC001 - 用例名称
- **前置条件**: ...
- **测试步骤**: ...
- **预期结果**: ...
- **优先级**: P0/P1/P2
- **自动化**: ✅/⏸️/❌
```

**示例参考**：
- `traveler-platform/test-cases/统计看板测试用例.md`

### 3. 执行测试

**自动化测试**：
1. 编写测试脚本（参考 `traveler-platform/`）
2. 运行测试套件
3. 生成测试报告

**手动测试**：
1. 按照测试用例步骤执行
2. 记录测试结果
3. 截图保存到 `assets/screenshots/`

### 4. 生成测试报告

**使用模板**：
- 参考 `traveler-platform/test-reports/第一轮测试报告-20260303.md`
- 包含：测试统计、通过/失败用例、缺陷列表、性能数据

**报告命名**：
- 格式：`第X轮测试报告-YYYYMMDD.md`
- 示例：`第一轮测试报告-20260303.md`

### 5. 更新项目索引

每次测试后，更新：
1. 项目README中的测试进度
2. 项目README中的缺陷统计
3. `PROJECT-INDEX.md` 中的统计信息

## 📋 测试用例编写规范

### 用例ID命名
- 格式：TC[模块][序号]
- 示例：TC001、TC002、TC003

### 优先级定义
- **P0（核心）**：核心功能，必须通过
- **P1（重要）**：重要功能，应该通过
- **P2（一般）**：一般功能，尽量通过

### 测试步骤
1. 每个步骤清晰明确
2. 包含具体操作和数据
3. 可重复执行

### 预期结果
1. 具体可验证
2. 包含数值和状态
3. 明确成功/失败标准

## 📊 测试报告编写规范

### 报告结构
1. **测试概况**：统计信息、通过率
2. **测试结果**：详细结果、通过/失败
3. **缺陷列表**：严重程度、状态
4. **性能数据**：加载时间、响应时间
5. **测试结论**：风险评估、下一步计划

### 报告频率
- **每日报告**：测试执行阶段
- **周报**：测试周期结束
- **最终报告**：项目验收

## 🔍 常见问题

### Q1: 如何选择测试工具？
**A**: 根据项目特点选择：
- Web应用：Playwright、Selenium、Cypress
- 移动应用：Appium、Espresso、XCUITest
- API测试：Postman、JMeter、RestAssured
- 性能测试：JMeter、Gatling、Locust

### Q2: 测试用例如何复用？
**A**: 
1. 设计通用测试用例模板
2. 提取公共测试流程
3. 参数化测试数据
4. 建立测试用例库

### Q3: 如何提高自动化覆盖率？
**A**: 
1. 优先自动化P0用例
2. 逐步扩展到P1、P2用例
3. 定期维护测试脚本
4. 持续集成自动化测试

### Q4: 知识库如何维护？
**A**: 
1. **每周**：检查项目完整性
2. **每月**：归档已完成项目
3. **每季度**：优化知识库结构
4. **持续**：提炼测试经验
